User guide

Documentation

A practical guide for preparing Excel backlog files and importing or updating Jira issues with controlled mapping, Description Builder, value transformations, validation and reporting.

Overview

Excel Backlog Importer for Jira helps teams create or update Jira issues from structured Excel workbooks. It is designed for project managers, analysts, Scrum Masters and Jira admins who receive backlog data in spreadsheets and need a reviewable workflow before Jira changes are made.

Before you start

  • Confirm the target Jira project and issue types.
  • Check that you have Jira permissions to create or update issues.
  • Prepare a workbook where each row represents a work item, update row or hierarchy-related row.
  • Use stable identifiers if you plan to update existing Jira issues later.

Preparing your Excel file

Use clear column names, one header row and consistent values. Avoid merged cells in the import area. Keep issue type, summary and parent/grouping columns explicit.

Supported file format

The product is built around Excel .xlsx workbooks and supports legacy .xls workbooks where available. CSV conversion is not required for the core Excel import workflow.

Mapping Excel columns to Jira fields

During import, map spreadsheet headers to Jira fields. Keep mappings focused on the fields you actually need. The same Excel column can be reused across multiple Jira fields when a workbook keeps several useful values in one source column.

Description Builder

Description can be mapped from a single column or built from multiple Excel columns. Description Builder combines selected columns into ordered sections, skips empty cells and can append unmapped workbook context while excluding common technical import columns.

Advanced value transformations

Each mapped field uses the full Excel value by default. Advanced value transformation is available when a user needs to extract part of a source cell. It supports regex extraction, optional result templates and preview before validation.

  • Use the first capture group, the full match or a result template such as $1 or ${1}.
  • Normalize supported Jira date fields using explicit input formats such as DD.MM.YYYY, YYYY-MM-DD, DD/MM/YYYY or MM/DD/YYYY.
  • Invalid regex, unsafe patterns, invalid result templates and invalid dates are blocked before import.

Working with custom fields

The app is designed to support practical Jira field families discovered from Jira metadata. Exact behavior depends on Jira configuration, field type, field screen availability and permissions.

  • Text, paragraph text, numbers, dates and date-time fields.
  • Single select, multi-select, checkbox-like fields, labels-like arrays and cascading select values.
  • User picker and multi-user picker fields using account ID, email address, display name or Name <email@example.com>.
  • Priority, components, fix versions, security level, sprint IDs and Assets object identifiers where Jira metadata supports safe mapping.

Time tracking

Time tracking has a dedicated mapper for original estimate and optional remaining estimate. Supported input styles include plain numbers interpreted by the selected unit, hours, working days, man-days and Jira duration text such as 1w 2d 3h.

Importing epics, stories and sub-tasks

Parent and sub-task behavior depends on Jira project configuration, issue type hierarchy, field screens and permissions. Use explicit issue type and parent columns, then validate the relationship before import. Sub-task rows require a parent reference from a default parent or a mapped parent column.

Updating existing issues

For update workflows, use a stable identity such as Jira Issue Key, External ID, Requirement ID or Backlog ID. Repeated uploads can update matching Jira issues instead of creating duplicate work where the import setup supports the match. Only mapped fields should be treated as part of the update workflow.

Saved setup and repeat imports

After successful validation, the app can save setup for the uploaded file name in the current Jira project. Restored setup can include sheet, header row, mappings, value transformations, Description Builder settings, duplicate strategy, estimate settings and locale/date settings.

Duplicate handling

Duplicate handling uses configured row identity and import metadata to help with repeated workbook uploads. Users can also choose an import mode that imports every selected valid row as new work when duplicate checking should be ignored.

Import history

Import metadata and reports help teams review what happened, support duplicate handling and troubleshoot later questions.

Exporting import reports

After an import, export the result report where available. Reports are useful for project records and post-import review.

Troubleshooting

  • Missing summaries usually mean the mapped Summary column is empty for selected rows.
  • Invalid select values usually mean the spreadsheet value does not match Jira project configuration.
  • User lookup issues can depend on Atlassian account visibility and permissions.
  • Regex transformation errors usually mean the pattern, result template or date input format needs adjustment.
  • Duplicate identity issues usually mean the selected identity column is missing, inconsistent or reused unexpectedly.
  • Hierarchy errors often come from unsupported parent-child relationships in the Jira project.

Limitations

  • Hierarchy behavior depends on Jira project configuration, issue type hierarchy and permissions.
  • Custom field support depends on the field type and Jira configuration.
  • Advanced value transformation is bounded regex extraction with preview, not a full spreadsheet formula engine or ETL system.
  • Trial and pricing are managed through the Atlassian Marketplace listing.
