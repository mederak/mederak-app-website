User guide

Documentation

A practical guide for project managers, analysts and Scrum Masters importing Excel backlogs into Jira Cloud.

Quick start

  1. Open the app from a Jira project.
  2. Upload a .xlsx workbook.
  3. Select the worksheet and header row.
  4. Map Excel columns to Jira fields.
  5. Choose duplicate handling, import mode and optional parent issue.
  6. Validate rows and import only selected valid rows.
  7. Review the report and export CSV or JSON if needed.

Creating new Jira issues from Excel

Use the standard import flow when rows in the spreadsheet should become new Jira issues. The app reads the selected worksheet, uses the header row you choose, lets you map Excel columns to Jira fields, validates the data and then creates only the selected valid rows.

This is useful for discovery workshop outputs, accepted commercial estimates, implementation task lists, client-provided requirements and initial project backlog setup.

Updating existing Jira issues from Excel

Excel Backlog Importer is not limited to creating new Jira issues. When the same Excel file is uploaded again, the app can recognize previously imported rows and update the matching Jira issues instead of creating duplicates.

For recurring updates, keep the same file name and use a stable row identity column such as ID, Feature ID, External ID, Requirement ID or Backlog ID. During import, choose an update mode when existing Jira issues should be refreshed instead of creating new ones.

Typical update scenarios include refined requirement descriptions after a client workshop, corrected time estimates after technical analysis, changed priorities or planned dates, and added labels, components or owner information.

Importing under a parent issue

You can import backlog items directly under a selected parent issue. This is useful when your backlog is organized by Epics, modules, delivery phases, work packages or change requests. Instead of importing everything into a flat project backlog, you can run multiple focused imports and place each Excel backlog exactly where it belongs in the Jira hierarchy.

The user selects a parent issue during import. Imported issues are then created as children of that parent where the relationship is supported by the Jira project configuration, issue type hierarchy and user permissions. Epics are a common example, but they are not the only possible parent type.

Recommended workflow for parent-based imports

  1. Prepare or choose the parent issue, for example an Epic, phase, package, module or change request.
  2. Prepare the Excel file with backlog items for that scope.
  3. Select the parent issue during import.
  4. Review mapped fields and validation results in the preview.
  5. Create the selected issues as children of the chosen parent.
  6. Repeat the process for later sheets, estimates, phases or work packages.

Interaction with update mode

Parent-based imports can also be used together with update mode. The first import can create issues under a selected parent. Later, the same Excel file can be uploaded again and the app can update matching child issues instead of creating duplicates.

Recommended Excel structure

The app does not require one rigid template, but import is easiest when every row represents one potential Jira issue and the file includes clear headers such as Summary, Description, Issue Type, Estimate, Assignee, Priority and External ID.

For repeat imports and updates, add a stable identity column such as ID, Feature ID, External ID, Requirement ID or Backlog ID. A stable identifier helps the app recognize rows that were already imported from the same source file.

Saved setup

When you import the same file name again, the app can reuse the saved setup: worksheet, header row, mapping, hierarchy settings and duplicate handling.

Duplicate handling

For recurring imports, choose a stable Record identity column such as ID, Feature ID or External ID. Duplicate handling helps avoid creating the same Jira issue twice and supports update mode by matching Excel rows to issues created earlier.

If you intentionally want every selected valid row to become a new Jira issue, choose Ignore duplicate checking.

Field mapping notes

  • Summary is required for Jira issue creation.
  • Description is converted to Jira-compatible document format.
  • Labels can be separated with commas, semicolons or new lines.
  • User fields can be resolved by account ID, email address or display name where Jira permits lookup.
  • Time tracking supports common hour/day formats and Jira duration text.
