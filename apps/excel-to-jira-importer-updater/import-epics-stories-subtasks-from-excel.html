Structured backlog imports

Import epics, stories and subtasks from Excel to Jira

Hierarchy is the part of Excel to Jira import where careful planning matters most. A flat list can become Jira issues row by row, but epics, stories, tasks and subtasks need valid parent relationships before Jira issues are created.

This page focuses on structured imports. For the full end-to-end process, start with the Excel to Jira import guide.

The hierarchy problem

Backlogs often arrive as grouped spreadsheet rows: epics at the top, stories beneath them, implementation tasks below those stories and notes in separate columns. That structure is obvious to a human reader but not always obvious to an import tool unless the workbook contains clear issue type, summary and parent reference data.

Jira configuration also matters. Available issue types, parent-child relationships, field screens and permissions can differ by project. A hierarchy that looks reasonable in Excel must still be valid in the target Jira Cloud project.

Common workbook patterns

There is no single universal spreadsheet layout for hierarchy. Useful patterns include one row per issue with a parent reference, one row per child item with parent data repeated in another column, or separate columns for levels such as Epic, Story and Sub-task. The right structure depends on how the workbook was produced and how the target Jira project is configured.

Current hierarchy modes

  • No hierarchy: each Excel row becomes one separate Jira issue.
  • Parent from an Excel column: one column identifies the parent row, and another column contains the matching row identity.
  • Multiple hierarchy columns: the workbook stores levels such as Epic, Story and Task in separate columns.
  • Existing Jira parent key: advanced flat imports can use a column with existing Jira parent keys when the project allows it.

How mapping helps

Mapping connects workbook columns to Jira fields and hierarchy settings. For structured imports, the important part is not only mapping Summary and Issue Type. You also need a consistent way to identify parent issues, child issues and any existing Jira parent keys used by the import.

If the workbook also has business context, acceptance criteria or implementation notes, use Description Builder to preserve that context in the Jira Description instead of trying to squeeze everything into a hierarchy column.

Parent validation before import

Validation should happen before Jira issues are created. The review step should make it clear which rows are planned for creation, which rows cannot be imported, and which parent relationships need correction. This matters because a wrong parent can be more disruptive than a missing optional field: it can put work in the wrong place and make backlog review harder.

  • Missing issue type or summary.
  • Parent keys, parent references or hierarchy paths that do not match the expected setup.
  • Repeated hierarchy paths that may create duplicate generated issues.
  • Unsupported parent-child relationships in the Jira project.
  • Invalid users, estimates, priorities, components or custom field values.

When to keep the import flat

Not every Excel backlog should become a hierarchy on the first import. If the parent relationships are unclear, if issue type names are inconsistent, or if the Jira project configuration is still being decided, a flat import under a selected parent issue may be safer. The team can then refine structure in Jira after the data is visible and validated.

FAQ

Can I import epics, stories and subtasks from one Excel workbook?

Yes, when the workbook structure and Jira configuration support the intended parent-child relationships and those relationships are validated before import.

Do my columns need exact names like Epic or Story?

No exact public column names are required by this guide. The workbook needs clear columns that can be mapped to issue types, summaries and parent references.

Can I attach new issues to existing Jira parents?

A workbook can include existing parent issue keys when the target Jira project supports those parent relationships and the import setup maps them correctly.

What should I validate before importing hierarchy?

Validate missing summaries, issue types, parent references, unsupported relationships, duplicate hierarchy paths and required Jira field values before creating issues.
